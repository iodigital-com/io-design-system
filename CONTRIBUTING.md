## Getting started

**Requirements:** Node >= 20.0.0

```bash
# Clone and install from repo root
git clone <repo-url>
cd io-design-system
npm ci
```

> `pnpm install --frozen-lockfile` also works if you prefer pnpm.

Start the development environment (Stencil watch + storefront at `localhost:3000`):

```bash
npm run dev
```

## Development workflow

### Running checks

```bash
npm run test              # Vitest unit tests
npm run type-check        # TypeScript (storefront)
npm run governance:check  # Validates workspace invariants
```

Or run the full quality-gate sequence in one command:

```bash
npm run build:quality-gates
```

This runs: `governance:check` → `build` → `test` → `type-check` → `build:storefront`

### Building

```bash
npm run build:components      # Build Stencil core only
npm run build:wrappers        # Build React + Vue + Angular wrappers
npm run build                 # Build components + all wrappers
```

> The framework wrappers (`components.ts`, `proxies.ts`) are **auto-generated by Stencil** — never edit them by hand. Re-run `npm run build:components` to regenerate after changing component APIs.

## Adding a new component

Follow these steps in order. All steps are required before a component can be merged.

### 1. Core component (io-components)

Create a directory at `io-components/src/components/io-{name}/` with these files:

| File | Purpose |
|------|---------|
| `types.ts` | TypeScript type unions for prop values |
| `io-{name}-styles.ts` | CSS-in-JS style generator — `var(--io-*)` tokens only |
| `io-{name}.tsx` | Stencil component definition |
| `io-{name}.spec.ts` | Default props / render tests |
| `io-{name}.{click\|change\|input}.spec.ts` | Event emission tests |
| `io-{name}.disabled.spec.ts` | Disabled-state tests |

**Rules for component implementation:**
- Styles must only use `var(--io-*)` tokens — never hardcode hex values or pixel sizes.
- Import styles from `-styles.ts`; never write CSS inline in `.tsx`.
- Focusable components: `shadow: { delegatesFocus: true }`.
- Form controls with IDs: generate IDs in `componentWillLoad()`, not in `render()`.
- Focus ring: use `var(--io-focus-ring-active)`, never `var(--io-shadow-focus-ring)` directly.
- New design values: add to `src/global/app.css` first, then reference via `var(--io-*)`.

### 2. Storefront pages (io-storefront)

Create `io-storefront/src/app/components/io-{name}/` with all five tab pages:

```
io-{name}/
├── layout.tsx          # PageHeader with title, description, tabs, category="Component"
├── page.tsx            # Redirect → /components/io-{name}/configurator
├── configurator/
│   └── page.tsx        # <Configurator> with story + propDefinitions
├── examples/
│   └── page.tsx        # Visual gallery with code in 4 frameworks
├── usage/
│   └── page.tsx        # Do's and Don'ts for designers/PMs
├── accessibility/
│   └── page.tsx        # Keyboard support, ARIA, WCAG checklist
├── api/
│   └── page.tsx        # Props / Events / Methods / Slots tables
└── io-{name}.stories.ts  # Story definitions and prop definitions
```

### 3. Register the tag (storefront)

Two files must be updated together or TypeScript will error:

**`io-storefront/src/utils/generator/generator.tsx`** — add to `IoTagNames` union:
```ts
export type IoTagNames =
  | 'io-button'
  | 'io-{name}'   // add here
  | ...
```

**`io-storefront/src/types/custom-elements.d.ts`** — add JSX interface:
```ts
'io-{name}': DetailedHTMLProps<
  HTMLAttributes<HTMLElement> & {
    // props here, kebab-case attribute names
  },
  HTMLElement
>;
```

### 4. Add to sitemap

In `io-storefront/src/sitemap.ts`, add to the Components section in alphabetical order:
```ts
{ label: '{Label}', href: '/components/io-{name}/configurator' },
```

> Do not add to sitemap until all five tab pages exist — a partial entry causes 404s.

### 5. Verify

```bash
cd io-components && npm run test && npm run type-check
cd ../io-storefront && npm run type-check
```

## Code conventions

### Tokens first

Never hardcode visual values. Every colour, spacing, radius, or shadow must reference a `var(--io-*)` token:

```ts
// Bad
color: '#0000D2';
padding: '16px';

// Good
color: 'var(--io-color-primary)';
padding: 'var(--io-space-4)';
```

### Naming

- Web Components: `io-{name}` (kebab-case tag, `Io{Name}` Stencil class)
- Framework wrappers: `Io{Name}` (PascalCase)
- Type unions: `Io{Name}{Prop}` (e.g. `IoButtonVariant`)

### TypeScript

- `@ts-expect-error` is not permitted as a workaround for custom element typing — add the correct type to `custom-elements.d.ts` instead.
- All public component props must have explicit TypeScript types.

### Accessibility

- WCAG AA minimum — 4.5:1 text contrast, 3:1 UI component contrast.
- Interactive elements: 44 × 44px minimum touch target.
- All interactive elements must be keyboard-accessible and focusable.
- Use semantic HTML before ARIA — ARIA only where native elements are insufficient.

### Tests

- Unit tests use Vitest + jsdom.
- `vi.mock()` must be at the top level of the spec file (not inside `beforeEach`).
- Every component requires at minimum: render test, event emission test, disabled-state test.

## Commit messages

Use [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <subject>
```

| Type | When to use |
|------|-------------|
| `feat` | New component or feature |
| `fix` | Bug fix |
| `refactor` | Code change that is not a fix or feature |
| `docs` | Documentation only |
| `test` | Adding or updating tests |
| `chore` | Build scripts, dependency updates, tooling |
| `style` | Formatting, whitespace (no logic change) |

**Scope** is optional but recommended — use the package or component name:

```
feat(io-button): add xl size variant
fix(io-tabs): resolve tabs.map TypeError when prop is a string
docs(readme): add Angular quick-start example
chore(deps): update @stencil/core to 4.44.0
```

## Pull request checklist

Before opening a PR, verify:

- [ ] `npm run build:quality-gates` passes with no errors
- [ ] New component: all five storefront tab pages exist
- [ ] New component: `IoTagNames` union and `custom-elements.d.ts` updated
- [ ] New component: added to `sitemap.ts` in alphabetical order
- [ ] No hardcoded design values — all via `var(--io-*)`
- [ ] No `@ts-expect-error` suppressions
- [ ] Commit messages follow the conventional commits format

## Quality gates

The full gate sequence must pass before any merge:

```bash
npm run build:quality-gates
```

Individual gates:

| Command | Checks |
|---------|--------|
| `npm run governance:check` | Workspace topology, curated file integrity, no legacy paths |
| `npm run build` | Stencil build + all framework wrapper builds |
| `npm run test` | Vitest unit tests |
| `npm run type-check` | TypeScript (storefront) |
| `npm run build:storefront` | Next.js production build |

> GitHub Actions CI is intentionally disabled during the current development phase. Run the gates locally before opening or updating a pull request.
